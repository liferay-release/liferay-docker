# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Purpose

This repo builds and publishes official Liferay Docker images to Docker Hub (`liferay/dxp`, `liferay/portal`, `liferay/base`, `liferay/jdk21`, etc.). It covers the full lifecycle: downloading release bundles, building images, testing, and pushing.

## Commands

### Running Tests

```bash
# Run all tests (detects what changed vs upstream/master and runs relevant suites)
./run_tests.sh

# Run a specific test file directly
./test_env_common.sh
./test_release_common.sh
./test_build_bundle_image.sh
./test_patching_tool_version.sh

# Show successful test results (default: only failures shown)
DISPLAY_SUCCESSFUL_TEST_RESULT=true ./run_tests.sh
```

### Building Images

```bash
# Build all images locally (no push)
./build_all_images.sh

# Build only a specific version
LIFERAY_DOCKER_IMAGE_FILTER=2026.q1.4-lts ./build_all_images.sh

# Build and push all images (multi-arch amd64+arm64)
./build_all_images.sh --push-all

# Build infrastructure images only (single-arch push)
./build_all_images.sh --push

# Build a specific image type
./build_base_image.sh
./build_jdk21_image.sh

# Build a bundle image from a specific URL
LIFERAY_DOCKER_RELEASE_FILE_URL=files.liferay.com/private/ee/portal/... ./build_bundle_image.sh push
```

### Release Notes / Versioning

```bash
./release_notes.sh get-version      # Print current version (e.g., "10.0.28")
./release_notes.sh display          # Show what the next version would be
./release_notes.sh commit           # Bump version + write changelog entry
./release_notes.sh fail-on-change   # Exit 1 if unreleased changes exist (CI gate)
```

Version history is tracked in `.releng/docker-image.changelog`.

## Architecture

### Image Dependency Hierarchy

```
ubuntu:questing
  └── liferay/base  (curl, tini, jq, liferay user)
        ├── liferay/jdk11, jdk21, jdk11-jdk8, jdk21-jdk11-jdk8
        │     └── liferay/dxp:<version>    (Liferay DXP)
        │     └── liferay/portal:<version> (Liferay Portal CE)
        ├── liferay/batch
        ├── liferay/jar-runner
        ├── liferay/job-runner
        ├── liferay/node-runner
        ├── liferay/nexus-publisher
        └── liferay/caddy, squid, noop, zabbix-*
```

The JDK parent is chosen dynamically based on product version: `jdk11-jdk8` for 7.0–7.3, `jdk11` for early 7.4, `jdk21` for 2024.Q3+.

### Bundle Image Build Pipeline

`build_bundle_image.sh` orchestrates these steps in order:
1. Copy `templates/bundle/` into a temp directory
2. Download and extract the Liferay bundle (ZIP/7z) into the temp dir
3. Patch the `FROM` line in the Dockerfile to select the correct JDK parent
4. Upgrade patching tool and optionally install a fix pack
5. Upgrade Tomcat if a newer compatible version is available; warm up Hypersonic DB
6. Download a trial DXP license (DXP only)
7. Run `docker build` / `docker buildx build` (multi-arch for push)
8. Run `test_bundle_image.sh` against the built image
9. Push to Docker Hub if requested
10. Remove temp build context

### `bundles.yml` — The Version Catalog

All Liferay bundle versions are defined here. `build_all_images.sh` reads it with `yq` to iterate versions.

```yaml
quarterly:
  2026.q1.4-lts:
    bundle_url: releases-cdn.liferay.com/dxp/2026.q1.4-lts/...
    latest: true              # at most one per DXP and one per Portal
    fix_pack_url: ...         # optional
    test_hotfix_url: ...      # optional
    test_installed_patch: ... # expected patch string after build
```

### Library Systems

There are two parallel utility libraries. New code in `release/` uses `lc_*`; image build scripts use `_common.sh`:

- **`_common.sh`** — older helpers used by build scripts (`download`, `make_temp_directory`, `prepare_tomcat`, `check_utils`, `log_in_to_docker_hub`)
- **`_liferay_common.sh`** — newer `lc_`-prefixed functions with download caching, colored logging, background process management, and step timing. **Do not edit this file unless you are editing the root copy** — it is forked into other repositories.
- **`_release_common.sh`** — version classification helpers (`is_quarterly_release`, etc.)

### Test Framework

Custom shell-based assertions in `_test_common.sh`:

```bash
function test_some_function {
    assert_equals "$(some_function arg)" "expected_output"
}
```

Failures are accumulated and printed at the end. File diffs use `diff --side-by-side`.

### Script Sourcing Pattern

Every script uses a guard so it can be both executed and sourced by test files:

```bash
function main {
    if [[ "${BASH_SOURCE[0]}" != "${0}" ]]; then
        return
    fi
    # ... logic
}
main "${@}"
```

### Dockerfile Template Substitution

JDK Dockerfiles use `@placeholder@` tokens replaced at build time via `sed`:

```bash
sed --in-place "s/@jdk_version@/${jdk_version}/g" "${TEMP_DIR}"/Dockerfile
```

### Multi-Architecture Builds

- Local builds strip the `--platform=${TARGETPLATFORM}` arg from Dockerfiles (single-arch)
- Push builds use `docker buildx build --platform linux/amd64,linux/arm64 --push` with the `liferay-buildkit` buildx builder

### Download Caching and Mirrors

- Build scripts cache to `downloads/`; `lc_download` caches to `~/.liferay-common-cache/`
- Liferay internal URLs are routed through `http://mirrors.${LIFERAY_DOCKER_MIRROR}.liferay.com/` (default mirror: `lax`)

### Key Environment Variables

| Variable | Description |
|---|---|
| `LIFERAY_DOCKER_RELEASE_FILE_URL` | Bundle URL for `build_bundle_image.sh` |
| `LIFERAY_DOCKER_IMAGE_FILTER` | Build only a specific version |
| `LIFERAY_DOCKER_HUB_TOKEN` / `_USERNAME` | Docker Hub credentials |
| `LIFERAY_DOCKER_REPOSITORY` | Image name prefix (default: `liferay`) |
| `LIFERAY_DOCKER_IMAGE_PLATFORMS` | Buildx platforms (e.g., `linux/amd64,linux/arm64`) |
| `LIFERAY_DOCKER_RELEASE_CANDIDATE` | Push to release-candidates repository |
| `LIFERAY_DOCKER_SLIM` | Build smaller image without Elasticsearch sidecar |
| `LIFERAY_DOCKER_MIRROR` | Download mirror location |
| `LIFERAY_COMMON_DEBUG_ENABLED` | Enable debug logging |

### `release/` Subsystem

Separate pipeline for building Liferay releases from source (not for Docker image builds). It clones `liferay-portal-ee` and `liferay-release-tool-ee`, builds the bundle, and publishes to GCS and Maven. Has its own test suite (`release/test_*.sh`).

### Required Tools

`docker`, `7z`, `curl`, `java`, `unzip`, `yq`, `jq`, `git`, `sed`, `sort`, `tr`, `gawk`; `docker buildx` with `liferay-buildkit` builder for multi-arch pushes.
